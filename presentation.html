why · the gap01 / 12
The problem

LLMs are unreliable at structured spec work.

SpecDD specs are binding contracts — must-do, must-not, owned files, dependencies, tasks, scenarios. Ask a model to chain-resolve, merge, or surgically edit them with raw file tools and it drifts: mangles bytes, miscounts inheritance, invents conflicts.

✕ Chain resolutionWalking root → target and merging inherited rules by hand is error-prone and order-sensitive.
✕ Byte-faithful editsFlipping one task state with Edit risks clobbering BOM, CRLF, indentation, comments.
✕ Conflict detectionMulti-owner overlap and forbidden-dependency checks need mechanical precision, not vibes.
concept · .sdd02 / 12
What is SpecDD

Specs that live next to the code they govern.

Small .sdd files sit beside their code. AI agents read them as binding development contracts — not documentation. Specs inherit: a feature spec sees every rule from the specs above it.

  • Must / Must not  — hard rules, inherited down the chain
  • Owns  — file authority, checked before edits
  • Tasks & Scenarios  — tracked, byte-faithfully updated
invoice.sdd
Spec: Invoice Service
Purpose: Issue + reconcile invoices

Must:
  - validate currency before persist
Must not:
  - call the ledger API directly
Owns:
  - src/billing/services/**

Tasks:
  [ ] #1 add refund path
  [x] #2 currency guard

design · principles05 / 12
Design principles

The code-better-than-LLM gate.

01
Earn your place
Every tool must do something the LLM does poorly, slowly, or inconsistently. Anything trivial with Read is not exposed.
02
One canonical parser
All tools share a single parser and a single ParsedSpec shape. No divergent interpretations of the same file.
03
Structured errors, never string panics
Every call returns { ok, error, message, details }. The agent can branch on error codes.
04
Explicit invocation over ambient magic
The workflow triggers on /specc — no silent heuristics. Tools are passive; the command is the active playbook.

internals · parser pipeline07 / 12
One canonical parser

Pure Python. No network. No surprises.

A staged pipeline turns raw bytes into a typed ParsedSpec — handling BOM, encoding, and binary detection, and preserving unknown sections with line numbers so the format can grow.

  • Position tracking  — every section carries start/end line
  • Byte-faithful tasks  — each task keeps indent + raw
  • Errors vs warnings  — hard fails vs soft anomalies, separated
parse_spec(path | content)
   │
   ▼
lexer        bytes → Line  # BOM, encoding
   ▼
sections     Lines → DetectedSection
   ▼
bullets      Must / Owns / Forbids …
tasks        [ ] #id  # keep raw
scenarios    name + steps
   ▼
ParsedSpec
   ▼
Ok(spec, warnings=[…]) | Err(code)
safety · the only write surface08 / 12
Modifying specs safely

update_task_status — the only write surface.

Flip a task state without touching another byte. Four guarantees make it safe under concurrent agents.

Byte-faithful preservationOnly the targeted state symbol changes. Line endings, BOM, indentation, multi-byte chars, comments, blank lines — all preserved exactly.
Atomic writeTemp file + os.replace. A partially-written file is never observable on disk.
Stale-file detectionRequires expected_content_hash (SHA-256). If disk drifted, fails with STALE_FILE instead of clobbering.
Cross-process lock + all-or-nothingPer-spec flock/msvcrt. Multi-update batches are whole-batch atomic — any bad id and the file is untouched.
